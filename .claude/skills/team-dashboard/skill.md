# Team Dashboard Orchestrator — 팀 업무 현황 대시보드

노션과 슬랙에서 업무 데이터를 수집하고, 담당자별/프로젝트별/팀별로 교차 분석하여 인터랙티브 HTML 대시보드를 생성하는 오케스트레이터.

## 실행 모드: 서브 에이전트

## 에이전트 구성

| 에이전트 | subagent_type | 역할 | 스킬 | 출력 |
|---------|--------------|------|------|------|
| notion-collector | notion-collector | 노션 업무 항목 구조화 수집 | notion-work-scan | `_workspace/01_notion_collector_work_items.md` |
| slack-scanner | slack-scanner | 슬랙 대화 구조화 수집 | slack-discussion-scan | `_workspace/01_slack_scanner_discussions.md` |

## 워크플로우

### Phase 1: 준비

1. 사용자 입력 분석:
   - 특정 팀, 프로젝트, 기간이 언급되었으면 해당 범위로 한정
   - 별도 지정이 없으면 전체 워크스페이스 탐색
2. 작업 디렉토리에 `_workspace/` 생성 (이미 있으면 이전 산출물 보존)

### Phase 2: 데이터 수집 (병렬)

**실행 방식:** 병렬

단일 메시지에서 2개 Agent 도구를 동시 호출:

| 에이전트 | 프롬프트 내용 | model | run_in_background |
|---------|-------------|-------|-------------------|
| notion-collector | 프롬프트 템플릿 A | opus | true |
| slack-scanner | 프롬프트 템플릿 B | opus | true |

**프롬프트 템플릿 A (notion-collector):**
```
당신은 notion-collector 에이전트입니다. .claude/agents/notion-collector.md의 역할 정의를 따르세요.

## 작업 지시
노션 워크스페이스에서 업무 관련 항목을 수집하세요.
.claude/skills/notion-work-scan/skill.md의 워크플로우를 따라 수행하세요.

검색 키워드: {사용자가 지정한 키워드 또는 "기본 키워드 사용"}
범위: {특정 팀/프로젝트 또는 "전체"}

## 중요
- 각 작업에서 반드시 추출: 작업명, 상태, 담당자, 팀, 프로젝트, 마감일
- 팀 구분: 기획팀, 개발팀, 아트팀 (이 외 팀이 발견되면 추가)
- 수집 결과를 _workspace/01_notion_collector_work_items.md에 저장하세요.
- 발견한 프로젝트명/팀원명/팀명 목록을 반환 메시지에 포함하세요.
```

**프롬프트 템플릿 B (slack-scanner):**
```
당신은 slack-scanner 에이전트입니다. .claude/agents/slack-scanner.md의 역할 정의를 따르세요.

## 작업 지시
슬랙에서 업무 관련 대화를 검색하세요.
.claude/skills/slack-discussion-scan/skill.md의 워크플로우를 따라 수행하세요.

검색 키워드: {사용자가 지정한 키워드 또는 "기본 키워드 사용"}

## 중요
- 의사결정, 액션 아이템, 블로커를 구분하여 추출하세요.
- 각 논의를 관련 프로젝트/팀/담당자와 연결하세요.
- 노션에 미등록된 것으로 보이는 작업도 식별하세요.
- 수집 결과를 _workspace/01_slack_scanner_discussions.md에 저장하세요.
```

**notion-collector 완료 후 보강 검색:**
notion-collector가 반환한 프로젝트명/팀원명을 추가 키워드로 사용하여, slack-scanner가 아직 실행 중이면 별도 서브 에이전트로 보강 슬랙 검색을 수행한다. 이미 완료된 경우 건너뛴다.

### Phase 3: 교차 분석

1. 두 에이전트의 산출물을 Read로 수집:
   - `_workspace/01_notion_collector_work_items.md`
   - `_workspace/01_slack_scanner_discussions.md`

2. **담당자별 매핑**: 각 담당자의 노션 작업 + 슬랙 논의를 연결

3. **프로젝트별 매핑**: 각 프로젝트의 전체 작업 + 관련 슬랙 논의 연결

4. **팀별 매핑**: 기획팀/개발팀/아트팀별 작업 현황 집계

5. **슬랙 핵심 논의 추출** (Phase 5 터미널 보고에 반드시 포함):
   - 의사결정 사항: 최근 확정된 결정 (최대 5건)
   - 액션 아이템: 담당자 → 내용 → 기한 (최대 5건)
   - 블로커/이슈: 내용 + 관련 프로젝트 + 심각도 (전수)
   - 슬랙 데이터가 없으면 "슬랙 데이터 없음 — 연결 소스 설정이 필요합니다" 메시지를 Phase 5에 전달

6. **갭 분석**:
   - 슬랙에서 논의 중이나 노션에 미등록된 작업 식별
   - 노션에 있으나 7일 이상 미업데이트된 작업 식별
   - 담당자 미배정 작업 식별

7. **핵심 수치 집계**:
   - 전체 활성 작업 수
   - 긴급/임박 작업 수 (3일 이내)
   - 지연 작업 수
   - 블로커 수
   - 미등록 작업 수

### Phase 4: HTML 대시보드 생성

대시보드 스펙은 `references/dashboard-spec.md`를 참조하여 HTML 파일을 생성한다.

**생성 절차:**
1. `references/dashboard-spec.md`를 Read로 로드
2. Phase 3의 분석 결과를 스펙의 데이터 구조에 맞춰 정리
3. 스펙의 HTML 구조 + 스타일 + 스크립트를 조합하여 단일 HTML 파일 생성
4. `_workspace/dashboard.html`에 저장

**HTML 파일 특성:**
- 자체 포함 (CDN 의존 없는 인라인 CSS/JS)
- 탭 전환: 팀별 / 프로젝트별 / 담당자별 / 갭 분석
- 요약 카드: 핵심 수치 한눈에
- 상태별 색상 코딩: 긴급(빨강), 진행중(파랑), 대기(회색), 완료(초록)
- 반응형 레이아웃 (모바일 대응)

### Phase 5: 주간 보고 출력

1. **터미널에 종합 주간 보고를 직접 출력한다** (가장 중요한 산출물):

   ```
   ═══════════════════════════════════════
    주간 업무 현황 보고
    생성일: YYYY-MM-DD
   ═══════════════════════════════════════

   ■ 핵심 수치
   ┌──────────┬──────┬──────┬──────┬──────┐
   │ 전체작업  │ 긴급  │ 지연  │블로커 │미등록 │
   │   N건    │ N건  │ N건  │ N건  │ N건  │
   └──────────┴──────┴──────┴──────┴──────┘

   ■ 팀별 현황
   ▸ 기획팀 (N건): 진행중 N / 대기 N / 긴급 N
   ▸ 개발팀 (N건): 진행중 N / 대기 N / 긴급 N
   ▸ 아트팀 (N건): 진행중 N / 대기 N / 긴급 N

   ■ 긴급/지연 항목
   [작업명] — 담당: OOO, D+N 지연
   [작업명] — 담당: OOO, D-N 임박

   ■ 슬랙 핵심 논의
   의사결정: [내용] (채널, 날짜)
   액션아이템: [담당] → [내용] (기한)
   블로커: [내용] (관련 프로젝트)

   ■ 갭 분석
   슬랙에서만 논의 중 (노션 미등록): N건
   7일 이상 미업데이트: N건
   담당자 미배정: N건

   ═══════════════════════════════════════
   상세 대시보드: _workspace/dashboard.html
   ═══════════════════════════════════════
   ```

   **슬랙 데이터가 없는 경우:**
   "■ 슬랙 핵심 논의" 섹션에 "슬랙 데이터 없음 — 연결 소스 설정이 필요합니다" 표시.
   나머지 섹션은 노션 데이터만으로 출력한다.

2. HTML 대시보드도 병행 생성 (Phase 4 유지)

3. `_workspace/` 디렉토리 보존 (중간 산출물 감사 추적용)

4. 대시보드 파일 열기 안내:
   - macOS: `open _workspace/dashboard.html`

## 데이터 흐름

```
사용자 입력 (키워드/범위)
       │
       ├→ [notion-collector] → work_items.md ──┐
       │     (담당자/프로젝트/팀 구조화)          │
       │                                        ├→ [오케스트레이터]
       └→ [slack-scanner] → discussions.md ────┘    ├ 교차 분석
                                                     ├ 갭 감지
                                                     ├ HTML 대시보드 생성
                                                     └ 터미널 요약
                                                        │
                                                        ↓
                                                 _workspace/dashboard.html
```

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| notion-collector 실패 | 1회 재시도. 재실패 시 슬랙 데이터만으로 부분 대시보드 생성 |
| slack-scanner 실패 | 1회 재시도. 재실패 시 노션 데이터만으로 대시보드 생성 (슬랙 섹션 "데이터 없음" 표시) |
| 양쪽 모두 실패 | 사용자에게 알리고 수동 검색 제안 |
| HTML 생성 실패 | 마크다운 보고서를 대체 산출물로 출력 |
| 데이터 충돌 (노션과 슬랙 정보 불일치) | 양쪽 출처를 병기하고 삭제하지 않음 |
| 팀/프로젝트 분류 불확실 | "미분류" 섹션에 배치, 원본 정보 보존 |

## 테스트 시나리오

### 정상 흐름
1. 사용자가 "업무 대시보드 만들어줘"라고 입력
2. Phase 1에서 기본 키워드/전체 범위로 설정
3. Phase 2에서 notion-collector와 slack-scanner가 병렬 실행, 각각 구조화된 산출물 생성
4. Phase 3에서 교차 분석: 담당자별/프로젝트별/팀별 매핑 + 갭 분석
5. Phase 4에서 HTML 대시보드 파일 생성
6. Phase 5에서 터미널 요약 + 파일 경로 안내
7. 예상 결과: `_workspace/dashboard.html` 생성 + 터미널에 핵심 수치

### 범위 지정 흐름
1. 사용자가 "개발팀 업무 현황 파악해줘"라고 입력
2. Phase 1에서 "개발팀" 범위로 한정
3. 수집 에이전트들이 개발팀 관련 데이터만 집중 수집
4. 대시보드가 개발팀 중심으로 생성 (다른 팀은 축약 표시)

### 에러 흐름
1. Phase 2에서 slack-scanner가 "슬랙 연결 소스 없음" 반환
2. 1회 재시도 후에도 동일 결과
3. 노션 데이터만으로 Phase 3~4 진행
4. 대시보드의 슬랙 관련 섹션에 "슬랙 데이터 미수집 — 연결 소스 설정 필요" 표시
5. 터미널에 슬랙 연동 설정 안내
