### Task 1.2: reconciliation.ts 신규 생성
**파일:** `src/lib/reconciliation.ts` (~150줄)

```typescript
// 핵심 타입
interface ReconciliationResult {
  item: WorkItem;           // 노션 작업
  conflict: boolean;        // 불일치 감지 여부
  conflictReason?: string;  // "슬랙에서 블로커 신호 감지" 등
  slackSignals: SlackSignal[];  // 관련 슬랙 메시지
  confidence: "high" | "medium" | "low";
}

interface SlackSignal {
  summary: string;
  category: SlackCategory;
  ts: string;
  channelName: string;
}
```

**매칭 로직 (Outside Voice 반영):**
1. **1단계: 채널 → 프로젝트** (기존 WeeklyReport.tsx에서 추출)
   - CHANNEL_ALIASES + normalize + 부분 문자열 비교
   - **WorkItem.project를 source of truth로 사용** (Eng Review Outside Voice #3A: inferProjectFromDB()와 다를 수 있음)
   - 채널 단위로 프로젝트에 귀속

2. **2단계: 메시지 → 작업 (수정됨)**
   - **1순위: 담당자 매칭** — Slack userId → 이름 → Notion assignee 일치
   - **2순위: 키워드 매칭** — 작업 제목에서 3글자 이상 핵심 키워드 추출, 담당자 매칭된 후보 내에서만 적용
   - Go/No-Go 게이트: spike 시 3건 미만이면 채널 단위 폴백

3. **3단계: 불일치 감지**
   - **Reconciliation 전용 블로커 키워드 (좁게):**
     - KR: "블로커", "막혔", "막힘", "지연", "진행이 안", "불가", "대기중"
     - EN: "blocked", "blocker", "delayed", "stuck", "waiting"
   - **제외 (흔한 단어, 분류용으로만 사용):** "이슈", "버그", "에러", "문제", "크래시"
   - **부정 패턴 제외:** "이슈 없음", "해결됨", "no blocker", "resolved", "문제 없음"
   - **해결 메시지 확인:** 같은 채널에서 블로커 메시지 이후 "해결", "resolved", "fixed" 메시지가 있으면 conflict 해제

4. **Confidence 레벨:**
   - high: 담당자 매칭 + 블로커 키워드 + 미해결
   - medium: 채널 매칭만 + 블로커 키워드
   - low: 키워드 매칭만
